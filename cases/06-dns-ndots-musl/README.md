# Case 06 — Intermittent DNS in Alpine container (`ndots` + musl resolver)

The ticket (#4640, P3) said: *"Our worker container intermittently can't resolve `api` — the service name in our compose. Same compose, same network, same image base for the worker and the api itself, but the worker hits NXDOMAIN about a third of the time. Restarting the container makes it work for a while. We're on Alpine 3.19."*

"Intermittent" was the word that did the most damage in this ticket — for the customer's interpretation, and almost for mine. Container DNS is rarely actually intermittent. It's almost always deterministic, but state-dependent in a way that looks random from outside. So the first thing was to rule out non-determinism. I exec'd into the worker and ran `for i in 1..5; do getent hosts api; done`. Same result every time. Not intermittent at this layer. Then the same loop with `dig api` — different answer. The two resolution paths were giving different verdicts on the same name.

That narrowed it. `cat /etc/resolv.conf` inside the container showed `nameserver 127.0.0.11`, `options ndots:0`, `search .` — Docker's embedded DNS, with `ndots:0` (which is what Docker writes for default networks; Kubernetes writes `ndots:5`). The interaction with musl is the part that bites: musl's `getent` does not consult the search list under `ndots:0` the same way glibc does, and Docker's embedded DNS maps short service names but only when the lookup arrives unqualified in a particular shape. If musl rewrites the query based on its own heuristics, the lookup pattern misses the embedded resolver entirely. `dig` goes straight to `127.0.0.11` over UDP and gets an answer; `getent` runs the lookup through musl's resolver path and depending on what musl decides about the trailing-dot situation, the embedded DNS may or may not see a query it knows how to handle.

The reliable workaround is to append the dot yourself: `api.` is fully qualified, musl's heuristics short-circuit on the trailing dot, and the embedded DNS gets a clean question. The customer's "third of the time" wasn't randomness — it was the same restart-time decision (which `search` entry musl picked, what state `nsswitch` was in, what `getent` cached) playing out across deploys. State-dependent failure that looks random from outside.

The smaller of the two factors is musl: Alpine's libc handles `search`, `ndots`, and trailing dots differently from glibc, so code that "just works" on a Debian base can fail on Alpine for the same hostname. The bigger one is Docker's embedded DNS, which is permissive for unqualified short names but only on custom networks and only when the query arrives in the expected shape. Stack the two together and you get a class of bug that doesn't reproduce on either Alpine-without-Docker or glibc-with-Docker, only on the intersection.

For the customer, two paths forward. The cheap one is to standardize on the FQDN form (`api.`) inside the worker — one search-and-replace through the codebase, lint rule to keep it that way, ship today. The expensive one is to move the worker off Alpine onto a glibc base like `python:3.12-slim`. The trade-off there is real and worth being honest about: the `python:3.12-slim` image is roughly 2–3× the size of `python:3.12-alpine` (the exact figures vary by point release; check `docker manifest inspect` or `docker images` for current values before sizing storage). For a worker container that pulls and starts once per deploy, the delta is negligible. For a fleet of short-lived job runners that pull on every cold start, it's a real cost in registry bandwidth and pull latency. The recommendation depends on which side of that line the customer is on. We told this customer the FQDN workaround was enough for them because their workers are long-lived; we'd flip the recommendation for someone running ephemeral jobs at scale.

The capture worth keeping is the side-by-side `getent` and `dig` output against `api` and `api.` — it makes the bug visible in two seconds to anyone reading `logs.txt`.

## Adjacent failure modes (not hit in this case, but the same pattern)

- **`nsswitch.conf` order on glibc.** Resolution order is configurable: `hosts: dns files` consults DNS first; `hosts: files dns` consults `/etc/hosts` first. A container that has a stale `/etc/hosts` entry plus `files dns` resolves to the wrong IP and never sees the DNS answer. Diagnostic: `cat /etc/nsswitch.conf` and `cat /etc/hosts` next to each other; the surprise is which one wins.
- **Kubernetes `dnsPolicy: ClusterFirst` writing `ndots:5`.** Pods get `options ndots:5` in `/etc/resolv.conf`, which means any name with fewer than 5 dots gets the search list applied first. A short private apex like `vault.internal` becomes `vault.internal.<namespace>.svc.cluster.local`, gets NXDOMAIN, then falls through. Symptom: 5 wasted lookups per resolution under the hood, latency spike under load. Fix: `dnsConfig.options: [{name: ndots, value: "1"}]` or rewrite calls to use trailing dots.
- **Long-running JVM with cached DNS.** Java's default DNS cache TTL on some JDK versions is `-1` (cache forever) for successful lookups. A container that resolves `db.internal` at startup keeps that IP forever, even after the database fails over and the DNS record updates. Symptom: app talks to a stale IP for hours after a failover; restart "fixes" it. Diagnostic: check `networkaddress.cache.ttl` in the JVM args or `java.security`; set to `60` or lower.

## Runtime notes

The bug shape above is Docker-specific. Docker injects an embedded DNS server at `127.0.0.11` and writes `ndots:0`; that's the combination that interacts badly with musl. On Podman, rootless networks use `aardvark-dns` at the network gateway (e.g. `10.89.0.1`) and `ndots` is not pinned, so both the short and fully-qualified forms resolve and the bug does not reproduce. The captured `logs.txt` reflects whichever runtime executed the script. The contract test asserts only the runtime-agnostic teaching point — that the fully-qualified form resolves reliably. The Docker-only signal (`nameserver 127.0.0.11`) appears in `logs.txt` when the runtime is Docker; absent on Podman.
